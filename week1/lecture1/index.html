    Week 1 — Lecture 1: Electronic Systems & Implementation¶

    Course & Lecture Info

    Course: Advanced Digital Design and Verification (ADDV)
    Instructor: Sneh Saurabh
    Lecture: Week 1, Lecture 1 — Introduction to Systems, Components, and Implementation

    Learning Goals¶

    • Define an electronic system and its elements.
    • Distinguish hardware, software, and other components in real systems.
    • Classify processors (fixed-behavior vs customizable) and related compute elements (coprocessors, microcontrollers, ASICs).
    • Recognize major memory categories.
    • Outline the hardware and software implementation flows.
    • Understand verification, manufacturing & test, and system integration & validation at a high level.

    1) What is a System?¶

    A system is a combination of elements that function together to produce the capability required to meet a need.

    Elements may include: hardware, software, equipment, facilities, personnel, and processes.

    Electronic systems: systems designed to sense, process, and manipulate electronic signals.
    Examples: mobile phone, computer, robot, television, EEG machine, etc.

    System/Element relationships (concept map)¶

    flowchart TD
    A[System Need] --> B[System]
    B --> C[Elements]
    C --> C1[Hardware]
    C --> C2[Software]
    C --> C3[Equipment & Facilities]
    C --> C4[Personnel]
    C --> C5[Processes]
    B --> D[Capabilities Produced]
    D --> E[Meets the Need]

    2) Components of Electronic Systems¶

    Systems often comprise numerous subsystems built with diverse technologies; the overall functionality is richer than individual parts.

    flowchart TD
    S["Electronic System"] --> H["Hardware"]
    S --> SW["Software"]
    S --> O["Other Components"]

    H --> H1["Processing"]
    H --> H2["Storage (Memory)"]
    H --> H3["Sensing"]
    H --> H4["Communication"]

    SW --> SW1["System SW (OS, drivers)"]
    SW --> SW2["Dev Tools (compiler, linker, debugger)"]
    SW --> SW3["Application SW (GP/Specialized)"]
    SW --> SW4["Firmware"]

    O --> O1["Packaging & PCBs"]
    O --> O2["Mechanical parts (hinges, keyboard)"]
    O --> O3["Power & Thermal"]
    • Hardware (electronic): compute, memory, sensors/actuators, I/O and comms.
    • Software: system software, development toolchain, application software, firmware.
    • Other components: packaging, PCB, mechanical parts, power delivery, etc.

    3) Hardware: Processor Types¶

    Instruction-set/architecture determines customizability.

    flowchart LR
    P[Processors] --> F[Fixed Behavior]
    P --> C[Customizable]

    3.1 Fixed-Behavior Processors¶

    • General-purpose: designed for broad computing requirements (e.g., x86, ARM cores).

    • Special-purpose: optimized for specific tasks (power, speed, area):

      • DSP processors

      • Image/vision processors

      • NPUs/TPUs (neural/tensor processing)

      • GPUs

    flowchart TD
    FB[Fixed Behavior] --> GP[General Purpose CPU]
    FB --> SP[Special Purpose]
    SP --> SP1[DSP]
    SP --> SP2[Image/Video]
    SP --> SP3[NPU/TPU]
    SP --> SP4[GPU]

    3.2 Customizable Processors¶

    • ASIP (Application-Specific Instruction-set Processors): add custom instructions for domain PPA gains.

    • Configurable cores: cache sizes, pipeline depth, functional units, interfaces.

    • Commercial examples include families like Xtensa and ARC-V with full toolchains.

    flowchart TD
    C["Customizable"] --> ASIP["ASIP (Extend ISA)"]
    C --> CFG["Configurable Core"]

    ASIP --> T1["Custom Instructions"]
    ASIP --> T2["Compiler/Toolchain Support"]

    CFG --> P1["Pipeline/Issue Width"]
    CFG --> P2["Cache Sizes/Hierarchy"]
    CFG --> P3["Custom Interfaces/Accelerators"]

    4) Coprocessors¶

    Definition: An auxiliary processor that supplements the main CPU for certain tasks (e.g., floating-point, graphics, signal processing, crypto, I/O, AI/vision).

    Why use them?

    • Optimized for specific algorithms → better performance/Watt.

    • Let the main CPU remain simpler (can reduce cost/power in some products).

    • Over time, popular coprocessor functions may migrate into CPUs.

    flowchart LR
    CPU[Main CPU] -- offload --> X1[FPU]
    CPU -- offload --> X2[GPU]
    CPU -- offload --> X3[AI/Vision]
    CPU -- offload --> X4[Crypto/Compression]
    CPU -- offload --> X5[Signal Processing]

    5) Microcontrollers (MCUs)¶

    One chip integrating CPU core(s) + memory + programmable I/O + peripherals.
    Often used in embedded/real-time systems (robotics, automotive, medical, IoT).

    • Typical features

      • GPIO (input mode: read sensors; output mode: drive LEDs/motors via power stages)

      • Timers/counters, watchdog

      • ADC/DAC, comparators

      • UART/SPI/I²C/CAN/etc.

      • On-chip Flash and SRAM

    flowchart TB
    subgraph MCU["Microcontroller (single IC)"]
      CORE["Processor Core"]
      MEM["Flash + SRAM"]
      BUS["Internal Bus"]
      TIM["Timers / PWM"]
      GPIO["GPIO"]
      ADC["ADC / DAC"]
      COMM["UART / SPI / I2C / CAN"]
    end

    CORE -- "fetch/exec" --> MEM
    CORE <---> BUS
    BUS <---> TIM
    BUS <---> GPIO
    BUS <---> ADC
    BUS <---> COMM

    GPIO --> OUT["Actuators/Motors (via drivers)"]
    GPIO --> IN["Sensors/Buttons"]
    ADC --> SENS["Analog Sensors"]

    Microcontroller vs SoC: an SoC is generally broader (may integrate multi-core CPUs, GPUs, memory controllers, etc.). An SoC can include MCU-like subsystems.

    6) ASICs¶

    Application-Specific Integrated Circuits — designed for a specific application rather than general use.

    • May integrate multiple processors, memories, accelerators, and peripherals.

    • Complex ASICs ≈ SoCs.

    Example application domains

    • Networking

    • Security

    • Voice processing

    • Video encode/decode

    • Storage and I/O controllers

    flowchart TB
    subgraph ASIC["ASIC / SoC (high-level)"]
      CPU1[CPU Cluster]
      ACC1[Domain Accelerator]
      MEM1[SRAM/TCAM/Cache]
      PERI[I/O Peripherals]
      BUS1[Interconnect/Fabric]
      PWR[Power/Clock/Reset]
    end
    CPU1 <--> BUS1
    ACC1 <--> BUS1
    MEM1 <--> BUS1
    PERI <--> BUS1
    PWR --> CPU1
    PWR --> ACC1
    PWR --> MEM1
    PWR --> PERI

    7) Memory Landscape¶

    flowchart TB
    M["Memory"] --> NVM["ROM (Non-Volatile)"]
    M --> VM["RAM (Volatile)"]
    M --> EXT["External Storage"]

    NVM --> PROM["PROM"]
    NVM --> EPROM["EPROM"]
    NVM --> FLASH["Flash"]

    VM --> SRAM["SRAM"]
    VM --> DRAM["DRAM"]

    EXT --> HDD["Hard Disk"]
    EXT --> ODD["Optical (CD/DVD)"]
    EXT --> USB["USB Flash"]

    Notes

    • NVM/ROM: retains data without power (PROM/EPROM/Flash).

    • RAM: fast working memory (SRAM for on-chip caches; DRAM for main memory).

    • External: system-level storage devices.

    8) Software Components¶

    Categories

    • System Software: OS, device drivers, boot/firmware.

    • Software Development Tools: compiler, assembler, linker, loader, debugger, profilers, CI.

    • Application Software: end-user applications (general-purpose or domain-specific).

    • Firmware: machine instructions embedded in devices (cameras, phones, NICs, printers, routers, remotes).

    flowchart LR
    SW[Software] --> SYS[System SW]
    SW --> DEV[Dev Toolchain]
    SW --> APP[Application SW]
    SW --> FWM[Firmware]

    SYS --> OS[Operating System]
    SYS --> DRV[Device Drivers]
    SYS --> BOOT[Boot / BIOS / UEFI]
    DEV --> COMP[Compiler/Assembler]
    DEV --> LINK[Linker/Loader]
    DEV --> DBG[Debugger/Profiler]
    APP --> GP[General Purpose]
    APP --> SPEC[Specialized/Domain]

    10) System Implementation — Bird’s-Eye View¶

    flowchart TD
    SF[System Functionality / Requirements] --> PART[Partitioning]
    PART --> HWF[HW Functionality]
    PART --> SWF[SW Functionality]
    HWF --> IC[IC Design & Fabrication]
    IC --> FAB[Fabricated ICs/SoCs/Peripherals]
    SWF --> SWDEV[Software Development]
    SWDEV --> EXE[Executables: Firmware, Drivers, Apps]
    FAB --> INT[System Integration, Validation & Test]
    EXE --> INT
    INT --> PROD[Product]

    Project-level actions

    1. Map functionality to HW vs SW.

    2. Build components
      A. HW design, verification, fabrication
      B. SW development

    3. Integrate and validate the combined system.

    11) Hardware Design Implementation¶

    Two major transformations:

    • Logic Synthesis: Convert RTL → equivalent netlist (in target library).

    • Physical Design: Convert netlist → layout (GDS) ready for fabrication.

    flowchart LR
    RTL["RTL"] --> SYN["Logic Synthesis"]
    SYN --> NET["Gate-level Netlist"]
    NET --> PD["Physical Design"]
    PD --> LAYOUT["Layout (GDS)"]
    LAYOUT --> FAB["Fabrication"]

    Typical physical design tasks

    flowchart TB
    A[Netlist] --> B[Chip Planning / Floorplanning]
    B --> C[Placement]
    C --> D[Clock Tree Synthesis]
    D --> E[Global Routing]
    E --> F[Detailed Routing]
    F --> G[ECO/Design Fixes]
    G --> H[Write GDS]

    12) Hardware Design Verification¶

    Goal: Ensure the design implements the intended functionality under constraints.
    Performed at multiple stages of the VLSI flow.

    • Functional Verification

      • Simulation-based (directed, random, constrained random, coverage-driven)

      • Formal methods (property checking, model checking)

      • Equivalence checking (e.g., RTL vs netlist)

    • Physical Verification

      • DRC, ERC, LVS

      • Signal integrity, IR drop, timing (STA), variation analysis

      • Sign-off checks

    flowchart TD
    V[Verification] --> FV[Functional Verification]
    V --> PV[Physical Verification]
    FV --> SIM[Simulation-based]
    FV --> FORM[Formal Methods]
    FV --> EQ[Equivalence Checking]
    PV --> DRC[DRC/ERC/LVS]
    PV --> STA[STA & Timing Reports]
    PV --> SI[SI/IR drop/EM]

    13) Hardware Manufacturing & Test¶

    Input: Final GDS from physical design.
    Output: Chips that pass production test.

    flowchart TD
    G[GDS] --> PRE[Pre-process & Mask Prep]
    PRE --> WAF[Wafer Fabrication]
    WAF --> DIE[Die Testing]
    DIE --> PKG[Packaging]
    PKG --> FT[Final Testing]
    FT --> CHIP[Chips Shipped]

    Device-level test loop (ATE)

    flowchart LR
    TP[Test Patterns] --> DIEU[Die Under Test]
    ER[Expected Responses] --> CMP{Match?}
    DIEU --> AR[Actual Responses]
    AR --> CMP
    ER --> CMP
    CMP -- Yes --> PASS[Test Pass / Accept]
    CMP -- No --> FAIL[Test Fail / Reject/Diagnose]

    14) Software Development Process¶

    flowchart TD
    PLAN["Planning & Requirement Analysis"] --> ARCH["Designing Architecture"]
    ARCH --> CODE["Implementation / Coding"]
    CODE --> TEST["Testing (manual & automated)"]
    TEST --> DEP["Deployment & Integration"]
    DEP --> MAINT["Maintenance (bug fixes, enhancements, support)"]

    • Planning: resources, schedules, functional and technical definitions, documentation.

    • Architecture: layers, modules, responsibilities, technology selection.

    • Implementation: compilers/interpreters, build systems, debuggers used to produce executables.

    • Testing: unit/integration/system/regression; automation & CI.

    • Deployment: delivery to target environment; integration with other components.

    • Maintenance: fixes, performance optimization, new features.

    15) System Integration & Validation¶

    Integration begins as components become available. It surfaces interface/integration issues early.

    • Validation highlights

      • Run actual OS/applications on fabricated silicon or accurate platforms.

      • Exercise system-level scenarios not covered in unit tests.

      • Find electrical issues (crosstalk, IR drop, thermal, variation).

      • Stress/soak tests and corner/abuse cases.

    flowchart TD
    FAB[Fabricated HW/Platform] --> INT[System Integration]
    EXE[SW: Firmware/Drivers/Apps] --> INT
    INT --> VAL[Validation]
    VAL --> PROD[Ship Product]

    Key Takeaways¶

    • Electronic systems combine hardware + software + other disciplines to meet a defined need.

    • Compute choices range from fixed-behavior CPUs/accelerators to customizable ASIPs.

    • MCUs integrate compute + peripherals on one die for embedded control; ASICs/SoCs tailor whole chips to applications.

    • The VLSI flow transforms RTL → netlist → layout while verification ensures correctness.

    • Manufacturing & ATE confirm defect-free parts; integration/validation demonstrates end-to-end behavior.
