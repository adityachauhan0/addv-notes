    Week 1 – Lecture 2: Electronic System Level (ESL) Design¶

    Course: Advanced Digital Design and Verification
    Instructor: Sneh Saurabh • Date: 14 Aug 2025

    TL;DR¶

    • Move up the abstraction stack (ESL) to fight SoC complexity.
    • Create an executable specification → analyze → partition HW/SW → implement → verify → iterate.
    • Use TLM, SystemC, VIPs, and HLS/ISS/virtual platforms to accelerate design + verification.

    Motivation¶

    Why RTL-first breaks in modern SoCs

    • More features, more HW+SW, more comms → time ↑, cost ↑, FoM compromises.
    • Strict time-to-market makes long RTL cycles risky.

    Counter-move

    • Higher-level abstraction (ESL) to boost design efficiency and parallelize HW/SW development.

    Effects vs Fix

    Pain ESL Relief
    Long design time Early modeling + architectural exploration
    Costly late changes Fast iteration at high abstraction
    FoM compromises Early analysis of power/area/perf trade-offs
    Verification bottlenecks Co-verification, VIPs, mixed-abstraction TBs

    What is ESL?¶

    A higher-level executable abstraction of the system that is above RTL, aimed at comprehension, exploration, and early verification.

    Expectations

    • Time-to-result ↓, FoM ↑, cost-effective, concurrent HW/SW development.

    Decisions & Deliverables at ESL¶

    • Architectural exploration/estimation/optimization

      • Concurrency: sequential / parallel / pipelined / multithreaded

      • Communication: bus / buffered / memory / point-to-point

      • Functional adequacy: algorithm choice, arbitration, data representation (fixed vs float)

    • HW–SW partitioning with fewer experiments

    • Parallel SW development before HW exists

    • IP & HW–SW integration

    • RTL generation

    • Faster verification (mixed abstraction, auto TB scaffolding)

    Specifications & Executable Specs¶

    What counts as an executable spec?

    • A runnable functional/behavioral model that exhibits intended behavior.
    • Tech: SystemC, TLM, SystemVerilog, MATLAB, XML, IP-XACT, SDL.

    Levels of executable spec¶

    flowchart TB
  AS["Architecture Spec\n(ISA, behaviors, features)"]
  DS["Design Spec\n(uarch: pipeline, caches, counts)"]
  TLM["Transaction-Level Modeling\n(bus transactions, timing options)"]

  AS --> DS --> TLM

    Good spec attributes

    • Complete & precise functional requirements + constraints
    • Avoid premature platform details
    • Evolves with refinement; supports mixing abstraction levels
    • Source of truth for requirements → implementation → verification

    Analysis & Modifications at ESL¶

    • Pre-partition: time/space/power/complexity/TTM trade-offs

    • Static: maintainability, usability, criticality, reliability

    • Dynamic: latency/throughput, arbitration/scheduling effects

    • Reqs for analysis: computation (fixed/float), comms (bandwidth)

    Partitioning (HW vs SW)¶

    flowchart TD
  E["Executable Spec"] --> C{"Criteria"};

  C -->|Perf target| HW1["HW Candidate"];
  C -->|Flexibility| SW1["SW Candidate"];
  C -->|Area/Power| HW2["Accelerator"];
  C -->|Safety/Security| HW3["Isolated IP"];
  C -->|Reuse| SW2["FW / Driver"];

  HW1 --> HWAs["HW Subsystem"];
  HW2 --> HWAs;
  HW3 --> HWAs;

  SW1 --> SWAs["SW Subsystem"];
  SW2 --> SWAs;

  HWAs --> IF["Defined Interfaces (TLM/AMBA/etc)"];
  SWAs --> IF;

    Decide also:

    • and type of processors (and potential ISA changes)¶

    • Memory org: ROM/SRAM/DRAM/…

    • Interconnect/communication scheme

    • Sub-partitioning across multiple CPUs and multiple HW teams

    Post-partitioning: Analysis, Debug, Verification¶

    • Analysis: performance, interfaces, area, cost, debugability

    • Debug: run SW on ISS; model HW with SystemC/HDL; ensure interface compatibility

    • Verification planning @ higher abstraction:

      • Coverage at TLM

      • Assertions/properties at the same level

      • Mixed-abstraction environments (e.g., TLM + RTL)

    • Iterate if intent not preserved.

    Software Implementation Models¶

    Waterfall (pre-ESL)¶

    • Linear, sequential; late feedback.

    Spiral (with ESL)¶

    • Iterative, risk-driven; SW & HW co-evolve.
    gantt
    dateFormat  X
    title Waterfall vs Spiral (conceptual)
    section Waterfall
    Requirements   :a1, 1, 2
    Design         :a2, 3, 2
    Implement      :a3, 5, 2
    Integrate/Test :a4, 7, 2
    section Spiral (overlapping)
    Prototype/Model(ESL)   :b1, 1, 1
    Risk Analysis          :b2, 2, 1
    Build/Refine           :b3, 2, 2
    Evaluate/Plan Next     :b4, 4, 1
    Next Spiral            :b5, 5, 3

    Applying ESL to Software¶

    • Generate SW from ESL or refined models (software synthesis) or hand-code guided by the model.

    • Observability into HW via models.

    • Scope

      • ISS: CPU regs/mem visible

      • Virtual Platform: full SoC (user I/F, storage, accelerators)

    • Exec platforms: workstation, FPGA-based emulation

    • Metrics: speed, memory footprint

    flowchart LR
  SSW["Software"];
  ISS["Hardware Model (ISS)"];
  VP(("Virtual Platform"));

  SSW <--> ISS;
  SSW --> VP;
  ISS --> VP;

    Debuggability¶

    Approach Pros Cons
    HW trace buffers Real silicon visibility Area/perf cost, security concerns
    ESL-based PC/data trace from models, no HW cost Model fidelity limits, discipline

    Hardware Implementation Paths¶

    • RTL to gates via synthesis.

    Options

    1. Extensible processors (custom instructions, SIMD/MAC) + vendor toolchains

    2. Co-processors (DSP/VLIW attached on bus)

    3. ASIC (HLS from C/C++/SystemC → RTL)

    4. FPGA (rapid prototyping; same HLS path)

    graph LR
    H[ESL/C/C++/SystemC] --> HLS[High-Level Synthesis]
    HLS --> RTL[RTL]
    RTL --> ASIC[ASIC Flow]
    RTL --> FPGA[FPGA Flow]

    Implementation Verification¶

    • Verify implementation preserves design intent (compare to post-partition model).

    • Speed tricks: subsystem-level focus, mixed abstraction, FPGA prototypes.

    • Pass/Fail notions

      • Positive: requirement satisfied (valid inputs)

      • Negative: robustness (invalid inputs)

    Verification IPs (VIPs)¶

    • Reusable verification components.
    flowchart LR
    VIP[Verification IP]
    VIP -->|Dynamic| Dyn[Stimulus, Scoreboard, Coverage]
    VIP -->|Static| Sta[Assertions/Properties]

    History & State¶

    Why SystemC?¶

    • Bridges SW/HW; models parallelism in familiar C++.

    • Any valid C++ program is valid SystemC (it’s an extension).

    • Heavy use of templates/macros/libs.

    SystemC “Hello, World!”¶

    #include <systemc.h>

SC_MODULE(Hello_SystemC) {
    SC_CTOR(Hello_SystemC) {
        SC_THREAD(module_thread); // register thread process
    }
    void module_thread() {
        SC_REPORT_INFO("Module Thread Saying", "Hello SystemC World!");
    }
};

int sc_main(int argc, char* argv[]) {
    Hello_SystemC HelloWorld_i("HelloWorld_i"); // create instance
    sc_start();                                  // start simulator (run to time 0 unless events)
    return 0;
}

    Compile (example)¶

    # from the directory with main.cpp
g++ -I<path_to_systemc_include> -L<path_to_systemc_lib> main.cpp -lsystemc -o hello_sc
./hello_sc

    Checklist before compiling

    • g++ installed
    • -I points to systemc/include
    • -L points to the lib directory (e.g., lib-linux64)
    • Link with -lsystemc
    • If runtime can’t find the shared lib, set LD_LIBRARY_PATH (Linux) or use rpath.
